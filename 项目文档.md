# MyTODO 待办事项管理应用设计与实现

## 1 绪论

### 1.1 选题背景与意义

在快节奏的现代生活中，人们面临着越来越多的待办事务，如何高效地管理日常任务成为提升个人效率的关键问题。传统的纸质清单方式存在易丢失、难以分类、无法提醒等局限性。随着智能手机的普及，基于移动端的待办事项管理应用成为解决这一需求的有效手段。

本课题设计并实现一款基于 Android 平台的待办事项管理应用——MyTODO。该应用支持任务的创建、编辑、删除、完成标记，提供分类管理、重要标记、"我的一天"快捷视图、循环任务自动生成等核心功能。采用 MVVM 架构模式与 Room 持久化框架，确保应用具备良好的可维护性、数据一致性和用户体验。

本项目的研究意义在于：实践 Android 应用开发的主流技术栈（MVVM + Room + Navigation Component），探索待办管理类应用的交互设计模式，并为后续功能扩展（如云端同步、智能提醒等）奠定架构基础。

## 2 需求分析与设计

### 2.1 需求分析

通过对现有待办管理类应用的调研与分析，本应用确定以下核心功能需求：

**（1）待办任务管理**
- 创建新待办任务，支持设置标题、截止时间、所属分类
- 编辑已有任务的所有属性
- 删除不需要的任务，支持确认提示
- 标记任务完成/未完成状态

**（2）分类管理**
- 预置分类：工作、个人、购物
- 用户可自定义创建新分类
- 删除分类时级联删除其下所有任务

**（3）快捷视图**
- "我的一天"：快速查看标记为"我的一天"的任务
- "重要"：查看所有标记为重要的任务
- "全部待办"：查看所有任务

**（4）循环任务**
- 支持设置任务按指定天数循环
- 完成循环任务后自动生成下一个截止时间的任务

**（5）界面交互**
- Material Design 3 风格界面
- Chip 筛选器切换"全部/未完成"视图
- 空状态提示

### 2.2 系统架构设计

本应用采用 **MVVM（Model-View-ViewModel）** 架构模式，整体分为四层：

```
┌─────────────────────────────────────┐
│            View Layer               │
│  MainActivity / Fragments / Adapters │
├─────────────────────────────────────┤
│          ViewModel Layer            │
│  CategoriesViewModel / TodosViewModel│
├─────────────────────────────────────┤
│         Repository Layer            │
│          TodoRepository             │
├─────────────────────────────────────┤
│           Data Layer                │
│  Room Database / DAOs / Entities    │
└─────────────────────────────────────┘
```

各层职责如下：

- **View 层**：负责 UI 渲染与用户交互，包含 Fragment、Adapter 和 XML 布局文件。Fragment 通过 ViewBinding 获取视图引用，通过 LiveData 观察数据变化并自动更新界面。
- **ViewModel 层**：持有 UI 所需数据，处理业务逻辑，管理 LiveData 的生命周期。通过 `AndroidViewModel` 获取 `Application` 上下文以创建 Repository。
- **Repository 层**：数据访问的统一入口，封装 DAO 操作，所有数据库操作通过 `databaseWriteExecutor` 在后台线程执行。
- **Data 层**：Room 数据库、DAO 接口和实体类，负责数据的持久化存储与查询。

页面导航采用 Navigation Component 实现，由 `MainActivity` 作为宿主 Activity，内嵌 `NavHostFragment`，通过 `nav_graph.xml` 定义页面跳转关系。Fragment 之间的数据传递通过 Navigation Safe Args 或 Bundle 实现。

### 2.3 数据库设计

本应用采用 Room 持久化框架（基于 SQLite）进行数据持久化存储，设计两张数据表：`todo_categories`（分类表）和 `todo_items`（待办任务表），通过外键建立一对多关联关系。

**todo_categories 表结构**

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 分类唯一标识 |
| name | TEXT | NOT NULL | 分类名称 |

**todo_items 表结构**

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 任务唯一标识 |
| title | TEXT | NOT NULL | 任务标题 |
| due_date | INTEGER | NULLABLE | 截止时间（毫秒时间戳） |
| category_id | INTEGER | FOREIGN KEY → todo_categories(id), ON DELETE CASCADE | 所属分类 |
| is_recurring | INTEGER | DEFAULT 0 | 是否循环任务 |
| recurrence_interval_days | INTEGER | DEFAULT 1 | 循环间隔天数 |
| is_completed | INTEGER | DEFAULT 0 | 是否已完成 |
| is_my_day | INTEGER | DEFAULT 0 | 是否加入我的一天 |
| is_important | INTEGER | DEFAULT 0 | 是否标记为重要 |
| created_at | INTEGER | NOT NULL | 创建时间（毫秒时间戳） |

`todo_items` 表通过 `category_id` 外键关联 `todo_categories` 表，并设置 `ON DELETE CASCADE` 级联删除策略——当删除某个分类时，该分类下的所有任务将自动删除。`category_id` 字段建有索引以优化查询性能。

数据库创建通过 `Room.databaseBuilder()` 构建单例实例，在 `RoomDatabase.Callback.onCreate()` 回调中通过 `SupportSQLiteDatabase.execSQL()` 直接执行 SQL 语句插入预置分类数据（工作、个人、购物），避免使用 DAO + Executor 方式导致的竞态条件。

### 3.2 数据模型层实现

数据模型层包含两个 Room 实体类，定义了应用的核心数据结构。

**TodoCategory 实体**

`TodoCategory` 对应 `todo_categories` 表，包含 `id`（主键，自动生成）和 `name`（分类名称）两个字段。提供无参构造函数供 Room 使用，以及带 `@Ignore` 注解的单参数构造函数供业务代码便捷创建实例。

**TodoItem 实体**

`TodoItem` 对应 `todo_items` 表，实现 `Serializable` 接口以便在 Fragment 间通过 Bundle 传递。主要字段包括：

- `title`：任务标题
- `dueDate`：截止时间，类型为 `Long`（可空），存储毫秒时间戳
- `categoryId`：外键关联分类
- `isRecurring` / `recurrenceIntervalDays`：循环任务配置
- `isCompleted`：完成状态
- `isMyDay` / `is_important`：快捷标记
- `createdAt`：创建时间戳

实体类通过 `@ForeignKey` 注解声明与 `TodoCategory` 的外键关系，通过 `@Index` 注解为 `category_id` 建立索引。

### 3.3 数据访问层实现

数据访问层由 Room DAO 接口和 Repository 类组成。

**TodoCategoryDao**

提供分类的 CRUD 操作：
- `getAllCategories()`：按 ID 升序查询所有分类
- `getCategoryById(id)`：按 ID 查询单个分类
- `insert()` / `update()` / `delete()` / `deleteById()`：增删改操作

**TodoItemDao**

提供待办任务的多维度查询与 CRUD 操作，包括：
- 按分类查询（全部/仅未完成）
- 查询"我的一天"、"重要"任务
- 按截止时间排序查询
- 插入、更新、删除操作

所有查询方法返回 `LiveData<List<TodoItem>>`，实现数据变化的自动通知。

**TodoRepository**

Repository 作为数据层的统一入口，封装了所有 DAO 操作。核心特点：

1. **线程管理**：所有写操作通过 `AppDatabase.databaseWriteExecutor`（4 线程池）在后台线程执行
2. **completeTodo 方法**：完成循环任务的核心逻辑——标记当前任务为已完成，若为循环任务且有截止时间，则自动创建新任务，新截止时间 = 旧截止时间 + 循环间隔天数
3. **单例数据库**：通过 `AppDatabase.getDatabase()` 获取 Room 数据库单例

### 3.4 界面层实现

#### 3.4.1 主界面设计

`MainActivity` 作为应用的唯一 Activity，采用 Material Design 3 的 `AppBarLayout` + `Toolbar` + `FloatingActionButton` 布局。内嵌 `NavHostFragment` 作为导航容器，通过 `NavController` 管理 Fragment 之间的跳转。

FAB 按钮根据当前页面动态切换功能：在分类页面点击弹出"添加分类"对话框，在待办列表页面点击弹出"添加待办"对话框。通过 `NavController.addOnDestinationChangedListener` 监听页面切换，自动控制 FAB 的显示/隐藏。

顶部导航栏标题随页面变化：分类页面显示"分类"，待办页面显示当前分类名称或快捷视图名称（"我的一天"/"重要"/"全部待办"）。

#### 3.4.2 添加/编辑待办对话框设计

`AddTodoDialogFragment` 继承 `DialogFragment`，通过 `newInstance(categoryId, item)` 静态工厂方法创建实例，`item` 为 `null` 时为添加模式，非空时为编辑模式。

对话框布局包含以下表单元素：
- **标题输入框**：`TextInputLayout` + `TextInputEditText`
- **截止时间选择器**：`MaterialCardView` 包裹 `TextView`，点击弹出 `DatePickerDialog`，选中日期显示为 `yyyy-MM-dd` 格式，原始时间戳存储在 View 的 `tag` 中
- **分类下拉框**：`TextInputLayout`（ExposedDropdownMenu 样式）+ `AutoCompleteTextView`，通过 `ArrayAdapter` 填充分类列表
- **循环任务开关**：`MaterialSwitch`，开启后显示循环间隔天数输入框
- **重要/我的一天开关**：两个 `MaterialSwitch`
- **删除按钮**：仅编辑模式下显示，点击弹出确认对话框

关键实现细节：
- `onCreateDialog` 中创建对话框后显式调用 `setupViews()` 初始化视图（`DialogFragment` 使用 `onCreateDialog` 时 `onViewCreated` 不会被调用）
- 截止时间字段通过 `MaterialCardView` 作为可点击容器，设置 `foreground="?attr/selectableItemBackground"` 提供点击涟漪反馈
- 保存按钮通过 `dialog.setOnShowListener` 重写点击事件，阻止对话框自动关闭，仅在数据验证通过后手动 `dismiss()`

#### 3.4.3 列表交互设计

**TodosAdapter** 继承 `ListAdapter`，利用 `DiffUtil` 实现高效的列表差量更新。每个列表项包含：
- `CheckBox`：完成状态切换
- 标题 `TextView`：完成时显示删除线 + 50% 透明度
- 截止时间标签：格式化显示 `yyyy-MM-dd`，无截止时间时隐藏
- 循环标签：显示"每N天"，非循环时隐藏
- 重要标记图标

交互逻辑：
- 点击复选框 → 调用 `onCompleteToggle` 回调
- 点击列表项 → 调用 `onItemClick` 回调，弹出编辑对话框
- 使用 `Chip` 组件切换"全部/未完成"筛选视图

**CategoriesAdapter** 继承 `RecyclerView.Adapter`（非 `ListAdapter`），因为需要混合展示固定头部、快捷入口和动态分类三种类型的列表项。通过 `getItemViewType()` 返回不同类型常量，在 `onCreateViewHolder` 中加载不同布局。

### 3.5 关键功能实现

**（1）循环任务自动生成**

当用户勾选完成一个循环任务时，`TodosFragment.onCompleteToggle` 调用 `viewModel.completeTodo(item)`。Repository 的 `completeTodo` 方法在后台线程中：
1. 将当前任务标记为已完成并更新数据库
2. 判断是否为循环任务（`isRecurring && dueDate != null && recurrenceIntervalDays > 0`）
3. 创建新任务，复制原任务的所有属性，新截止时间 = 旧截止时间 + 循环间隔天数的毫秒值
4. 插入新任务到数据库

由于 Room 的 LiveData 机制，列表会自动刷新，用户可在列表中看到新生成的下一个循环任务。

**（2）分类级联删除**

通过 Room 的 `@ForeignKey(onDelete = CASCADE)` 注解实现。当删除一个 `TodoCategory` 时，SQLite 自动删除 `todo_items` 表中所有 `category_id` 指向该分类的任务记录，无需在业务代码中手动处理。

**（3）数据库预填充**

在 `AppDatabase` 的 `RoomDatabase.Callback.onCreate()` 回调中，直接通过 `SupportSQLiteDatabase.execSQL()` 执行 INSERT 语句插入预置分类数据。使用 `db.execSQL()` 而非 DAO + Executor 方式，避免与 Room 的创建事务产生竞态条件导致重复插入。

**（4）ViewModel 数据联动**

`TodosViewModel` 使用 `Transformations.switchMap()` 将 `categoryId` 的变化转换为对应的待办列表 LiveData。当用户切换分类时，`categoryId.setValue()` 触发 `switchMap` 重新执行查询，返回新分类的任务列表，Fragment 观察到数据变化后自动更新 RecyclerView。

## 5 总结与展望

### 5.1 工作总结

本项目完成了一款基于 Android 平台的待办事项管理应用 MyTODO 的设计与实现。主要工作包括：

1. **架构设计**：采用 MVVM 架构模式，结合 Room + LiveData + Navigation Component，构建了清晰的四层架构（View → ViewModel → Repository → Data），实现了关注点分离和数据驱动的 UI 更新。

2. **核心功能实现**：完成了待办任务的创建、编辑、删除、完成标记等基础功能，以及分类管理、重要标记、"我的一天"快捷视图、循环任务自动生成等增强功能。

3. **技术要点攻克**：解决了 `DialogFragment` 中 `onCreateDialog` 与 `onViewCreated` 的生命周期问题、`MaterialCardView` 内子视图的触摸事件拦截问题、Room 外键级联删除策略、数据库预填充的竞态条件等多个技术难点。

4. **用户体验优化**：实现了 Material Design 3 风格界面，提供了点击涟漪反馈、删除确认对话框、空状态提示、DiffUtil 高效列表更新等交互细节。

### 5.2 不足与展望

当前应用仍存在以下不足，也是后续迭代的优化方向：

1. **功能层面**：缺少任务提醒/通知功能，无法在截止时间前推送提醒；不支持任务拖拽排序；缺少子任务（Checklist）功能。

2. **数据层面**：数据仅存储在本地 SQLite 数据库，不支持云端同步与多设备共享；未实现数据导出/导入功能。

3. **架构层面**：`CategoriesAdapter` 未采用 `ListAdapter` + `DiffUtil`，在数据量较大时可能影响性能；部分 ViewModel 中的查询逻辑可进一步优化。
